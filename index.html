Clean code

Code needs to
  • Do the right thing
  • Run on available resources
  • Be maintainable
everything is tradeoffs clean code is about code that is maintainable if you're never going to change it, it doesn't need to be clean also, sometimes, the need for performance is going to override the need for maintainability most code is read far more often that it is written though, and in these cases, it's worth taking the extra time to make it maintainable also, it's easier to take care of the first two if you've taken care of the third one

Consistency & automation

Consistency increases the signal to noise ratio 

some code with lots of inconsistend bits in it
brains are good at patterns and noticing differences consistency makes it easier to concentrate on the important parts
Automation makes consistency easier
- code formatters - linters - static analysis tools
format doesn't matter, consistency does (if you really hate it, use your own format and reformat before push) linting is a mixture of formatting and actual coding practices - find a a base set of rules and extend them run the linter with autofix on! static analysis is a lot more complicated & expensive - work out if its good for you & your team (also isn't strictly automation cos it won't fix things for you)

Naming things

be consistent with names
Don't use - synonyms - alternate spellings - alternate punctuations
folder/directory, notification/message/alert colour/color, authorise/authorize variableNames, ClassNames, file-names, CONSTANT_NAMES (or whatever schema you prefer, just pick one)
be descriptive...
- don't use acronyms or abbreviations auth -> authn or authz? (although i'll use it to mean both) an ambigious abbreviation... - do use more words logoutButton is better than logout something where adding an adjective is helpful
...but not redundant
doLogin -> login theUser -> user, currentUser accountsObservable -> accounts/accounts$ login(loginEmail, loginPassword) -> login(email, password)

functions

- do one thing - no side-effects, no flags (don't do unexpected things!) - not have a tonne of parameters - not be too complex: nesting, complex ternaries - consist of a single level of abstraction - should be short - language idioms, but not clever code

Classes/modules

- should have a single concern - changes in a single location

why should you care? - code is easier to understand (easier to read, but also acts more expectedly) - less bugs - better performance

let, const & var
  1. Don't use var
  2. Use const if you can
  3. Use let if you must
What have I got against var? 

if (temperature < desiredTemperature) {
	turnOnHeater();
	var message = 'Warming up!';
} else {
	var message = 'Everything is awesome';
}

console.log(message);
var is _function scoped_ let & const are _block scoped_
ok, but why prefer const over let?
Functional programming paradigm It's easier to understand the code if the variable references don't change If it changes, it's a new thing - store it in a new variable
Gotcha: const only fixes the reference - you can still change the properties
this is something you can fix with a linter
